Threat Cloud
Threat Cloud is PanGuard's anonymous threat-intelligence network. It is off by default — until you turn on Collective Defense below, nothing leaves this machine. When you do enable it, your endpoint uploads anonymized threat signatures (no code, no file contents). New detection rules ship in the open ATR package — Guard checks once a day and tells you when an upgrade is available. It never auto-pulls rule logic from the network, so a compromised relay can't change what your engine enforces.

Flywheel
Your endpoint detects a threat → Threat Cloud aggregates it anonymously → it strengthens the open ATR ruleset → you get the improvements through pga upgrade (a daily check flags when new rules are out).

Collective defense

Off by default. When you turn it on, every attack Guard blocks on this machine can become a new ATR rule that protects everyone — and you get the community's new rules back faster. More sensors means quicker detection for all of us.
What turning it on does for you
  • Your blocked patterns help draft new open ATR rules.
  • You receive community rule updates sooner.
  • You can turn it off again at any time.
What is shared (and what is never shared)
  • Shared: the matched rule ID(s), attack category + MITRE technique, a coarse country region, and a truncated IP (last two octets zeroed).
  • Never shared: your prompts, code, file contents, secrets, file paths, hostname, or username.
  • A stable random install ID links your own submissions (pseudonymous) — raw samples never leave this machine.

Settings
Configure how Guard runs on this endpoint.

Guard Mode

Three modes. Learning builds a baseline and only logs. Report-only detects and records everything but never acts (recommended for new deployments). Protection actively blocks — but only using the responses armed in your enforcement policy (by default that's nothing destructive: it detects and notifies). Saving applies to new activity right away; it won't interrupt actions already in progress.

Semantic layer (Layer C) Off · optional

An optional second opinion. Layers A and B (the always-on ATR pattern rules and heuristics) do the blocking. Layer C adds your own LLM to read flagged events in plain language and catch novel attacks the deterministic layers miss. It is advisory only — it flags events for your review and can never block on its own. It runs only on events A or B already flagged, not on every action, so a cloud model typically costs a few cents a month, and a local model is free.
Path 1 — Local model (free & fully private)
  • Runs a model on this machine with Ollama.
  • No API key, no cost — nothing leaves your computer.
  • Best if you want maximum privacy.
Path 2 — Cloud model (bring your own key)
  • Use an OpenAI or Anthropic key you already have.
  • Typically a few cents a month (flagged events only).
  • Your key is sent only to that provider — never to PanGuard.
How to connect
Run this one command in your terminal. It walks you through picking a model and, for a cloud model, prompts for your key privately. You never type the key into this dashboard.
pga guard ai
When it finishes, restart Guard with pga guard restart — this panel will then show Layer C as connected.
How your API key is protected
  • Typed privately in the terminal. The key is read with the screen echo off, so it never shows on screen, never lands in your shell history, and is never passed as a command argument.
  • Never entered in the browser. This dashboard deliberately has no field for your key — a localhost web page is a weaker place to handle a secret than your terminal.
  • Encrypted at rest. Stored at ~/.panguard/llm.enc with AES-256-GCM, file permissions 0600 (only you can read it).
  • Locked to this machine. The encryption key is derived from this machine and user, so a copied file is useless anywhere else.
  • Sent only to your provider. Your key goes only to the model provider you chose. It is never sent to PanGuard.
Turn it off / remove your key
Turns Layer C off and deletes the stored config (including any key) from this machine.
pga guard ai --remove

Threat Cloud Endpoint

The server Guard talks to for Threat Cloud. Most people should leave this blank — Guard uses the default (tc.panguard.ai). Change it only if you run your own private or air-gapped relay: when Collective Defense is on, anonymized signatures and the daily rule-update check go to whatever URL is set here. Your prompts, code, and keys are never sent regardless.

Dashboard Port

The TCP port this dashboard listens on (localhost-only). Change it with pga config --dashboard-port <port>, then restart Guard.

Alerts

You are already set up. Every threat Guard detects shows up right here in this dashboard — on the Overview and Threats tabs. No configuration needed.
Send alerts somewhere else too? (optional)
You can also push each alert to a chat or inbox you already watch. Pick one below, paste its line into ~/.panguard-guard/config.json (or run pga setup for a guided walk-through), then restart Guard. Secrets are typed into that file, never into this dashboard. Heads-up: any channel you add sends alert summaries to that service (Slack's servers, your email provider, or your webhook URL). Alerts always stay visible here regardless.

Enforcement Policy — what Protection mode is allowed to do

This list is read-only here — an "armed" response only runs automatically when Guard Mode is set to Protection; in Learning or Report-only nothing automatic happens. To change what's armed, edit enforcementPolicy in ~/.panguard-guard/config.json (or run pga setup) and restart Guard.

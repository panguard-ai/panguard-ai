隱私是 Panguard Threat Cloud 的核心設計原則。所有威脅資料離開你的機器前都會匿名化，不收集任何遙測，一個 config flag 就能完全離線跑。

## 匿名化

### IP 遮罩

所有 IP 在提交前做 /16 匿名化，最後兩個 octet 歸零：

| 原始 IP        | 匿名化 IP     |
| -------------- | ------------- |
| `192.168.1.50` | `192.168.0.0` |
| `10.0.42.100`  | `10.0.0.0`    |
| `203.0.113.50` | `203.0.0.0`   |

保留了威脅情報需要的網段資訊，但不可能辨識出特定主機。

### 會分享的資料

| 資料                 | 分享 | 匿名化方式                    |
| -------------------- | ---- | ----------------------------- |
| 攻擊者來源 IP        | 有   | /16 遮罩（後兩個 octet 歸零） |
| 攻擊類型跟技術       | 有   | 本來就是通用分類，不需匿名化  |
| MITRE ATT&CK ID      | 有   | 公開分類法，不需匿名化        |
| 信心分數             | 有   | 不需匿名化                    |
| 檔案 hash（malware） | 有   | 只傳 hash，不傳檔案內容       |
| 時間戳               | 有   | 四捨五入到最近的整點          |
| 標籤跟分類           | 有   | 本來就是通用型                |

### 絕對不分享的資料

| 資料                 | 狀態         |
| -------------------- | ------------ |
| 你的機器 IP          | **絕不傳送** |
| hostname 或機器 ID   | **絕不傳送** |
| 內部 IP              | **絕不傳送** |
| 檔案內容             | **絕不傳送** |
| log 條目或原始事件   | **絕不傳送** |
| 使用者憑證           | **絕不傳送** |
| config 細節          | **絕不傳送** |
| 掃描結果             | **絕不傳送** |
| 使用者名稱或帳號名稱 | **絕不傳送** |

## 零遙測

Panguard 不收集任何使用資料：

- 沒有使用分析
- 沒有對外送的 crash report
- 沒有功能追蹤
- 沒有授權回撥（初次啟動除外）
- 沒有第三方分析 SDK
- 沒有瀏覽器指紋
- 沒有 session 錄影

## 資料保留

| 項目       | 預設       | 說明                     |
| ---------- | ---------- | ------------------------ |
| IoC 保留期 | 90 天      | 超過 90 天的指標自動清除 |
| 提交 log   | 僅本機     | 存在你的機器上，永不上傳 |
| Feed cache | 每小時更新 | 外部 feed 資料的本機快取 |

## 退出（離線模式）

Threat Cloud 分享可以完全關掉，適合氣隙或隱私敏感的環境：

```bash
# 在 Guard 裡關掉 Threat Cloud
panguard guard config --threat-cloud disabled

# 或用環境變數
export PANGUARD_THREAT_CLOUD=disabled
```

關掉之後：

- 不會向任何 Threat Cloud 實例提交資料（公開或私有都不會）
- Guard 繼續用本機偵測（ATR 規則、baseline）
- 威脅情報查詢只用上次 sync 的本機 feed cache
- 不會建立任何連到 Threat Cloud endpoint 的網路連線
- 其他功能全部正常

<Info>
  離線模式會降低偵測能力，因為你拿不到集體威脅情報。折衷方案是在你自己的網路內跑一台私有 Threat
  Cloud 實例。
</Info>

## 稽核 Log

所有向 Threat Cloud 的資料提交都會在本機記 log：

```bash
# 看提交 log
cat ~/.panguard/threat-cloud/submissions.log
```

每筆 log 記錄：

| 欄位        | 說明                        |
| ----------- | --------------------------- |
| `timestamp` | 什麼時候提交的              |
| `data`      | 實際送出去的匿名化資料      |
| `endpoint`  | 收到資料的 Threat Cloud URL |
| `status`    | HTTP response 狀態          |

讓你隨時都能驗證你的機器到底送出了什麼。

## 自架私有實例

要最大隱私的話，自己跑一台 Threat Cloud：

```bash
# 啟動私有實例
panguard threat start --port 8080 --api-key your-key

# agent 指過去
panguard guard start --threat-cloud http://your-private-server:8080
```

自架的好處：

- 所有威脅資料留在你的網路內
- 保留期、存取權、刪除策略你自己掌控
- 外部 feed sync（ThreatFox、URLhaus 等）照樣吃得到
- 跨 agent 關聯在你的機群內運作
- 沒有任何資料離開你的網路邊界

## GDPR 合規

Panguard Threat Cloud 依 GDPR 原則設計：

| 原則             | 做法                                   |
| ---------------- | -------------------------------------- |
| **資料最小化**   | 只收集威脅關聯需要的最少資料           |
| **目的限制**     | 資料專門用於威脅情報                   |
| **儲存限制**     | 可設定保留期（預設 90 天）搭配自動清除 |
| **被遺忘權**     | 自架實例可完全控制資料刪除             |
| **隱私內建設計** | 匿名化在資料離開 agent 前就做好了      |

<Tip>
  資料主權要求嚴格的組織，把自架 Threat Cloud
  跟關閉公開分享搭配使用。組織內有集體情報，但沒有任何資料離開你的網路。
</Tip>
